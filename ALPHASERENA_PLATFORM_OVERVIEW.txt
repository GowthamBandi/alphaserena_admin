 PART 1 — WHAT ALPHASERENA IS (one paragraph)
==============================================================================
AlphaSerena is a multi-tenant B2B (and soon B2C) fitness SaaS. You (the founder)
sell a subscription to fitness ORGANIZATIONS (gyms, yoga studios, personal-training
businesses, sports academies — "Comeback Sports", etc.). Each organization runs its
own gym inside the platform: it creates trainers (staff), manages clients (members),
and builds workout/diet plans. Members get their own mobile app to follow their
plans, track progress, and chat with their trainer. Everything runs on ONE shared
Firebase backend. Money flows in two layers (see Part 4).

The brand was previously called "TrainersHQ" and is migrating to "AlphaSerena".

 PART 3 — ROLES: WHO DOES WHAT
==============================================================================
SUPER ADMIN (you, the founder) — uses alphaserena_admin (web)
  • God-mode READ across the entire platform.
  • Creates the Tier-1 subscription plans gyms buy + discount coupons.
  • Approves / warns / blocks organizations (moderation).
  • Watches revenue, growth, expiring subscriptions, pending approvals.
  • Does NOT run any gym's daily operations.

ADMIN = GYM OWNER — uses trainersHQ (mobile)
  • Signs up, gets approved by you, buys a Tier-1 subscription.
  • Creates trainers (staff), manages clients (members).
  • Builds the content library: exercises, food, workout/diet/weekly plans.
  • Can only operate while approved + subscription active (else read-only).

TRAINER = STAFF inside one gym — uses trainersHQ (mobile)
  • Created by their gym owner (via a Cloud Function).
  • Works with assigned clients: assigns plans, writes notes, chats.
  • Bound to their org; same "can operate" gate as the org.

CLIENT = GYM MEMBER — uses alphaserena (mobile)
  • Follows assigned workout/diet plans, tracks progress, chats with the trainer.
  • (Future) buys the gym's own membership (Tier-2).
  • Today: a DATA RECORD managed inside trainersHQ; the app exists but isn't wired.

 PART 4 — THE MONEY MODEL: TWO SUBSCRIPTION TIERS (critical)
==============================================================================
TIER 1 — PLATFORM SaaS  (Founder  ->  Organizations)
  • YOU create plans in alphaserena_admin (collection: subscription_plans).
  • A gym owner buys one inside trainersHQ to unlock the app + get usage LIMITS
    (max trainers / clients / workout plans / diet plans / workouts).
  • Payment via Razorpay; verified SERVER-SIDE by a Cloud Function that derives the
    limits + duration FROM THE PLAN DOC (never trusts the client), then writes
    isSubscriptionActive + planExpiry + subscriptionLimits onto the gym's admin doc.
  • THIS IS LIVE/BUILT on the catalog side (Subscriptions screen) + verified schema.

TIER 2 — GYM MEMBERSHIPS  (Organization  ->  Clients)
  • Each gym would create its OWN membership plans for its members.
  • Members buy them inside the alphaserena client app.
  • NOT BUILT YET — no data model exists for it. Proposed (to design):
      membershipPlans/{id}         org-owned (adminId), the gym's member plans
      clientSubscriptions/{id}     a member's active membership + expiry
      + a Cloud Function to verify the member's payment & activate it.

 PART 5 — THE SHARED BACKEND (Firebase)
==============================================================================
PROJECT: trainershq-f5ded   ·   Auth + Firestore + Storage + (Cloud Functions).

IDENTITY & ACCESS COLLECTIONS
  master_admins/{uid}   Super admins (you). Membership here = founder access.
  admins/{uid}          Gym owners. docId == auth uid. Key fields:
                          name, email, phone, organizationName,
                          role, status (pending|active|warning|blocked),
                          statusReason, statusUpdatedAt, statusUpdatedBy,
                          subscription (map), subscriptionLimits {maxAdmins,
                          maxTrainers, maxClients, maxWorkoutPlans, maxDietPlans},
                          planName, planExpiry, isSubscriptionActive,
                          approvedBy, trainerIds[], clientIds[], timestamps.
  trainers/{uid}        Staff. docId == auth uid. name, email, phone,
                          specialization, experience, status (active|inactive),
                          assignedBy (owner uid), clientIds[], timestamps.

MEMBERS & RELATIONS
  clients/{id}                  Members (no login yet). name, phone, email, gender,
                                age, goal, adminId (org), trainerId (assigned),
                                status, notes, timestamps.  CANONICAL — not "clints".
  client_plan_assignments/{id}  Links a plan to a client (workout|diet).
  workout_notes_clients/{id}    Per-client trainer notes.
  chats/{clientId}/messages/{id}  Per-client chat thread (trainer <-> client).

TRAINING CONTENT (org-owned via adminId)
  exercises/{id}            name, muscleGroup, instructions, videoUrl, adminId.
  foodDatabase/{id}         name, serving, calories, protein, carbs, fat, adminId.
  workoutPlans/{id}         name, items[{exerciseId, sets, reps}], adminId, createdById.
  dietPlans/{id}            name, items[{foodId, qty, calories}], adminId, createdById.
  weeklyWorkoutPlans/{id}   Mon–Sun day -> workoutPlan schedule, adminId.

SUBSCRIPTION & BILLING
  subscription_plans/{id}   Tier-1 catalog (FOUNDER creates). Canonical schema:
                              title, duration (str), months (num), price (INR),
                              oldPrice?, points[] (features), isActive, order,
                              limits { trainers, clients, workoutPlans, dietPlans,
                              workouts }  <-- NESTED MAP (trainersHQ reads this).
                              (This app also writes flat maxTrainers/... for its own
                              back-compat; both are kept in sync by the model.)
  coupon_codes/{id}         Discount codes (super admin only). code, type
                              (percent|flat), value, isActive, expiresAt, description.
  admin_payments_history/{id}  Payment records written by the verify CF.

ORGANIZATION
  organizationProfiles/{adminId}  Public gym profile: name, logoUrl, about,
                              address, phone, email, instagram, website.

NOTE on a fixed data drift: the founder console used to read `users` (for members)
and `subscriptions` (for revenue) and write plans with FLAT limits. These were
WRONG. They are corrected: members = `clients`, revenue = `admin_payments_history`,
and plans now carry the nested `limits` map the rest of the platform expects.

 PART 6 — SECURITY RULES (firestore.rules, in the trainersHQ folder)
==============================================================================
The rules file lives in trainersHQ/firestore.rules and governs the shared backend
for ALL apps. Deploy from the trainersHQ folder:
    firebase deploy --only firestore:rules

KEY IDEAS
  • Role checks:
      isSuperAdmin()  = custom token claim role=='super_admin'  OR  a master_admins
                        doc exists. (Claim is preferred; set by set_super_admin.js.)
      isAdmin()       = an admins/{uid} doc exists.
      isTrainer()     = a trainers/{uid} doc exists; trainerAdminId() = their org.
  • status vocabulary on admins: pending | active | warning | blocked.
  • "CAN OPERATE" gate (orgCanOperate): an org may CREATE/EDIT data only when
        status not in (pending, blocked)  AND  isSubscriptionActive == true.
      -> An expired OR blocked org can still READ its data (see it + the notice),
         but cannot WRITE. This is what makes a trainer go read-only and the owner
         see a "Buy subscription" prompt. (Button visibility is app UI; the rules
         guarantee no writes slip through.)
  • TENANT ISOLATION: each gym can only read/edit its OWN content (exercises, food,
    plans, clients, assignments, notes, chats), matched by adminId. (Fixed an earlier
    leak where any gym owner could read/edit every gym's content.)
  • FOUNDER MODERATION: a super admin may update ONLY the moderation fields on an
    admin doc (status, statusReason, statusUpdatedAt, statusUpdatedBy, updatedAt).
    Subscription/limits/role are never client-writable.
  • SERVER-ONLY writes (Cloud Functions via Admin SDK, which bypass rules):
    create admins, create trainers, activate/renew a subscription.
  • subscription_plans: any signed-in user may READ (gyms browse to buy); only the
    founder may WRITE.   coupon_codes: founder only.
  • FALLBACK: super admin has god-mode READ on everything (powers the console
    dashboards); everything else denied; all writes default to locked.

DEPENDENCY: the operate-gate trusts the isSubscriptionActive boolean — a scheduled
Cloud Function must flip it to false when planExpiry passes (and planExpiry should be
stored as a Timestamp).

 PART 7 — CLOUD FUNCTIONS (functions/, TypeScript, in the trainersHQ folder)
==============================================================================
BUILT (in trainersHQ; compile clean):
  registerAdmin                 Creates admins/{uid} with safe defaults (pending,
                                zero limits). Prevents self-elevation.
  setAdminStatus                (super admin) approve/block/reactivate an admin.
  setRoleClaims                 (super admin) set a custom role claim.
  createTrainer                 (admin) provisions a trainer server-side (Auth user
                                + trainers doc + link + claim); enforces active
                                subscription + trainer limit.
  setTrainerStatus              (admin) activate/deactivate a trainer.
  previewCoupon                 (admin) validate a coupon, return discount.
  createRazorpayOrder           (admin) load plan + apply coupon + create order.
  verifyAndActivateSubscription (admin) verify Razorpay signature, derive limits +
                                months FROM THE PLAN DOC, write subscription state +
                                payment history.
  Secrets: RAZORPAY_KEY_ID, RAZORPAY_KEY_SECRET.

RECOMMENDED / NOT YET BUILT
  • A scheduled function to expire subscriptions (flip isSubscriptionActive when
    planExpiry passes) — REQUIRED for the operate-gate to be fully correct.
  • createHQ/staff helpers, refunds/reconciliation — future.
  • Tier-2 membership purchase verification — future (Part 4).

ONE-OFF SCRIPT (in alphaserena_admin/scripts/)
  set_super_admin.js   Bootstraps the founder: finds/creates the auth user, sets the
                       {role:'super_admin'} custom claim, writes master_admins/{uid}.

 PART 9 — APP 1: trainersHQ (the organization app)  [feature-complete]
==============================================================================
Architecture: Flutter + GetX (state, DI, named routes). The SessionController is the
routing brain: it watches auth, resolves the user (admins -> trainers ->
master_admins), and routes automatically (login / pending / blocked / subscription /
dashboard / super-admin). See trainersHQ/PROJECT_BRIEF.txt + CLAUDE.md for specifics.

SECTION 0 — Foundation & design system: theme tokens + reusable widgets,
  FsCollections constants, Cloud Functions backbone, firebase config, base rules.
SECTION 1 — Auth & identity: admin sign-up (registerAdmin CF), unified login, the
  SessionController routing brain, screens splash/login/register/pending/blocked.
SECTION 2 — Subscription & billing: plan catalog UI, Razorpay checkout wired to the
  CFs (server-verified), payment history, coupon preview.
SECTION 3 — Limits & usage: UsageController is the single source of truth for usage
  vs plan limits; LimitGuard blocks over-limit creates with an upgrade prompt.
SECTION 4 — Trainer management: admin creates trainers via createTrainer CF
  (server-side Auth provisioning so the admin stays logged in); list/detail/activate.
SECTION 5 — Client management: canonical clients collection; role-aware CRUD (admin
  sees all org clients, trainer sees assigned); create/edit/detail.
SECTION 6 — Workout & diet system: exercise library (with video upload), food
  database, workout/diet plans (composed from those), plan assignment to clients,
  per-client workout notes.
SECTION 7 — Chat: per-client thread chats/{clientId}/messages; admin + the client's
  trainer post; built so the client app can join the same thread later.
SECTION 8 — Organization profile & settings: public gym profile (logo upload), My
  Account (dark mode, sign out).
SECTION 9 — Super admin / platform: an in-app super-admin console (approve/block
  admins + coupons). NOTE: this overlaps the alphaserena_admin web console — decide
  which is canonical (the web console is the fuller founder surface).
SECTION 10 — Security & hardening: locked-down admins/trainers docs; weekly plans.
SECTION 11 — Polish & launch: main.dart boots the rebuilt app; analyze clean.
DEFERRED in trainersHQ: push notifications, app icon/splash, store metadata,
  refunds/reconciliation, the customer (client) app.

 PART 12 — CROSS-APP CONTRACTS & KNOWN GAPS
==============================================================================
  • Collection names are SHARED — never invent parallel/renamed collections. Use the
    canonical names (clients, admin_payments_history, workoutPlans, dietPlans, …).
  • Subscription plan schema: the nested `limits` map + title/months is the contract
    between the console (writer) and trainersHQ/the CF (readers). Keep both in sync.
  • Subscription gate: depends on isSubscriptionActive being kept accurate -> needs
    the scheduled expiry CF.
  • Client <-> auth linkage: undefined until the client app is wired (Part 11.3).
  • Tier-2 membership model: does not exist yet (Part 4).
  • Super-admin surface overlap: trainersHQ in-app console vs the web console.
  • Brand migration TrainersHQ -> AlphaSerena: in progress (client app still "Fitopia").

 PART 16 — GLOSSARY
==============================================================================
  Tenant / Org   = one gym (an admin + its trainers + clients + content).
  Super admin    = you, the founder; god-mode read + catalog + moderation.
  Operate-gate   = the rule that an org can only WRITE while active + subscribed.
  Custom claim   = a flag baked into the auth token (role=super_admin) — can't be
                   forged by the client; enforced in rules + functions.
  Tier-1         = founder -> organizations (the SaaS subscription).
  Tier-2         = organization -> clients (gym memberships; not built yet).
  Canonical      = the agreed single name/shape for a collection or field; never fork.
